# 需求智能体设计方案

基于现有 **AI 用例编写智能体**（`apps.ai_testcase`）的架构分析，给出「需求智能体」的几种实现方案，涵盖：产品（竞品分析、PRD、需求完善）、开发（需求分析、技术方案）、测试（需求分析、功能点梳理）。

---

## 一、现有 AI 用例智能体架构摘要

| 维度 | 实现方式 |
|------|----------|
| **模型** | Kimi K2.5（Moonshot API），支持 thinking、流式、多模态 |
| **客户端** | `KimiClient`，OpenAI 兼容接口，同步/异步 + 流式 |
| **Prompt** | `prompts.py` 按场景拆分：生成用例、模块重生成、分维度评审、采纳评审 |
| **数据** | `AiTestcaseGeneration`：需求摘要、result_json、xmind、source_files、token |
| **接口** | SSE 流式：`generate-stream`、`regenerate-module-stream`、`review-stream`、`apply-review-stream` |
| **多模态** | 支持上传 PDF/图片，`FileProcessor` 提取文本 + 图片 base64 入 prompt |

需求智能体可复用：**同一套 Kimi 客户端、FileProcessor、SSE 流式、配置（KIMI_*）**，主要新增「任务类型 + 角色化 Prompt + 结果结构」。

---

## 二、需求智能体能力矩阵

| 角色 | 能力 | 输入 | 输出形态 |
|------|------|------|----------|
| **产品** | 竞品分析 | 自家产品描述 + 竞品名/链接/文档 | 结构化报告（维度对比、优劣势、建议） |
| **产品** | 产品方案设计(PRD撰写) | 创意/目标用户/约束 | PRD 文档（目标、用户故事、功能列表、非功能需求） |
| **产品** | 需求 PR 完善 | 原始需求描述/初版 PRD | 完善后的 PRD（补全字段、澄清歧义、补充验收标准） |
| **开发** | 需求分析 | 需求文档/PRD | 分析报告（功能拆解、技术影响面、风险与依赖） |
| **开发** | 技术方案设计 | 需求/PRD + 现有技术栈说明 | 技术方案（架构、接口、存储、排期建议） |
| **测试** | 需求分析 / 需求理解 | 需求文档/PRD | 测试视角的分析（可测性、风险点、边界） |
| **测试** | 功能点梳理 | 需求文档/PRD | 功能点清单（模块/功能点/验收要点，可与现有用例结构对接） |

---

## 三、方案一：单应用多任务类型（与现有用例智能体并列）

**思路**：新建 `apps.ai_requirement`，一个应用内按「任务类型」区分能力，共用一套模型与流式接口，不同任务用不同 system prompt + 输出结构。

### 3.1 模型选型

- **主模型**：继续用 **Kimi K2.5**（与用例智能体一致，便于复用配置与客户端，长文档、思考模式适合 PRD/技术方案）。
- **可选**：在配置中增加 `REQUIREMENT_AGENT_MODEL`，便于后续切到 **DeepSeek-V3** 或 **GPT-4o** 做对比（成本/质量）。

### 3.2 智能体设计

- **一个“需求智能体”**，通过参数 **角色 + 任务类型** 选能力：
  - 角色：`product` | `dev` | `test`
  - 任务类型：如 `competitive_analysis` | `prd_draft` | `prd_refine` | `requirement_analysis` | `tech_design` | `test_requirement_analysis` | `feature_breakdown`
- **Prompt 设计**：每个 (角色, 任务类型) 对应一套 system prompt + 输出 JSON/Markdown 规范（见下方示例）。
- **流式**：与现有 `generate-stream` 一致，SSE 流式返回，最后解析为结构化结果并落库。

### 3.3 数据模型（示例）

```python
# ai_requirement/models.py
class AiRequirementTask(models.Model):
    ROLE_CHOICES = [('product', '产品'), ('dev', '开发'), ('test', '测试')]
    TASK_TYPE_CHOICES = [
        ('competitive_analysis', '竞品分析'),
        ('prd_draft', 'PRD撰写'),
        ('prd_refine', '需求PR完善'),
        ('requirement_analysis', '需求分析'),
        ('tech_design', '技术方案设计'),
        ('test_requirement_analysis', '测试-需求分析'),
        ('feature_breakdown', '功能点梳理'),
    ]
    role = models.CharField(max_length=20, choices=ROLE_CHOICES)
    task_type = models.CharField(max_length=40, choices=TASK_TYPE_CHOICES)
    requirement_input = models.TextField()  # 需求/PRD/描述
    status = models.CharField(...)  # generating / success / failed
    result_json = models.JSONField(null=True)  # 结构化结果
    result_md = models.TextField(null=True)   # 若输出为长文档
    source_files = models.JSONField(null=True)
    # prompt_tokens, completion_tokens, error_message 等与 ai_testcase 类似
```

### 3.4 接口设计

- `POST /api/ai_requirement/run-stream/`  
  Body: `role`, `task_type`, `requirement`, `files?`, `use_thinking?`  
  Response: SSE，事件类型可复用现有（start / chunk / done / error），`done` 时带 `data`（结构化）或 `result_md`。

### 3.5 优缺点

- **优点**：实现快，与现有用例智能体风格统一；一个应用、一套配置、易维护。
- **缺点**：所有能力共用一个模型，无法按任务选“更便宜”或“更专”的模型。

## 四、方案二：多智能体（产品 / 开发 / 测试 三个智能体）

**思路**：同一应用内定义三个“智能体”，每个智能体有固定角色和若干任务，便于在 UI 上按角色切换，且可为不同智能体配置不同模型。

### 4.1 模型选型

- **产品智能体**：长文档生成多（PRD、竞品报告）→ **Kimi K2.5** 或 **DeepSeek-V3**（长上下文、思考）。
- **开发智能体**：技术方案需要强推理 → **Kimi K2.5** 或 **GPT-4o**。
- **测试智能体**：与现有用例智能体衔接，偏结构化（功能点梳理）→ 可与用例智能体共用 **Kimi K2.5**，或单独设 **轻量模型**（如 Kimi 小模型）做“需求理解+功能点梳理”，降低成本。

### 4.2 智能体设计

- **产品智能体**  
  - 任务：竞品分析、PRD 撰写、需求 PR 完善。  
  - System prompt 统一为“产品经理角色”，再按 `task_type` 分支具体指令和输出格式。
- **开发智能体**  
  - 任务：需求分析、技术方案设计。  
  - System prompt 统一为“技术负责人/架构师”，再按任务类型分支。
- **测试智能体**  
  - 任务：需求分析（测试视角）、需求理解、功能点梳理。  
  - System prompt 统一为“测试分析师”，输出可与现有用例的模块/功能点结构对齐，便于后续一键生成用例。

每个智能体一个“入口” API，例如：  
`POST /api/ai_requirement/product/run-stream/`  
`POST /api/ai_requirement/dev/run-stream/`  
`POST /api/ai_requirement/test/run-stream/`  
Body 中再带 `task_type` 与输入内容。

### 4.3 优缺点

- **优点**：角色清晰，便于按角色配置不同模型和 Prompt；前端可做“产品/开发/测试”三个工作台。
- **缺点**：Prompt 与路由稍多，需避免重复代码（可共用 base prompt + 按角色/任务拼接）。

## 五、方案三：统一入口 + 任务路由 + 可选多模型

**思路**：一个统一入口，根据「任务类型」路由到不同 Prompt 和**可选**不同模型，平衡灵活性与成本。

### 5.1 模型选型

- **默认**：**Kimi K2.5**（与现有一致）。
- **可配置**：在配置/数据库中为任务类型指定模型，例如：
  - 竞品分析、PRD 撰写、技术方案 → `kimi-k2.5`（长文档、思考）
  - 功能点梳理、需求理解 → `kimi-k2.5-8k` 或其它低成本模型（若 Moonshot 提供）
  - 未来可接入 **DeepSeek**、**OpenAI**，在配置中写 `REQUIREMENT_AGENT_MODEL_BY_TASK` 映射。

### 5.2 智能体设计

- **单入口**：`POST /api/ai_requirement/run-stream/`，Body 含 `task_type`（必填）+ `requirement` + `files` + `options`（如 use_thinking、model_override）。
- **路由表**：  
  `task_type` → (system_prompt, output_schema, 建议模型)  
  例如：`competitive_analysis` → 竞品分析 prompt + 固定 JSON 结构；`feature_breakdown` → 功能点梳理 prompt + 与用例智能体一致的模块/功能点 JSON。
- **客户端**：抽象一层 `RequirementAgentClient`，内部按配置选择 `KimiClient` 或其它 OpenAI 兼容客户端，保持 views 只关心“任务类型 + 流式”。

### 5.3 输出结构建议（与现有体系对齐）

- **功能点梳理**：输出与 `ai_testcase` 的 `result_json` 的模块/功能点部分一致，便于后续“从功能点生成用例”或关联到 `FunctionalRequirement`。
- **需求分析（开发/测试）**：统一为 JSON（如：功能拆解列表、风险点、依赖、可测性建议），便于前端展示和后续步骤消费。
- **PRD/竞品报告**：可 JSON（结构化）+ 或 长 Markdown 存 `result_md`，便于编辑与版本管理。

### 5.4 优缺点

- **优点**：扩展性强，可逐步加任务类型和模型；统一入口对前端简单；可按任务控制成本与质量。
- **缺点**：配置与路由稍复杂，需要约定好 `task_type` 枚举与文档。

## 六、推荐与实施顺序

- **短期**：采用 **方案一（单应用多任务类型）**，与现有 `ai_testcase` 并列，复用 Kimi、FileProcessor、SSE，只新增 `ai_requirement` 应用 + 角色/任务类型 + 各任务 Prompt。这样能最快上线产品/开发/测试的 7 类能力。
- **中期**：在方案一基础上，为“需求智能体”增加 **可选模型配置**（即引入方案三的“按任务选模型”），并统一入口为单一 `run-stream`，便于后续接 DeepSeek/OpenAI。
- **长期**：若需要强角色化体验，再拆成 **方案二** 的多智能体 API 与前端工作台，底层仍可共用同一套 Prompt 与模型配置。

---

## 七、Prompt 设计要点（各任务类型）

1. **竞品分析**  
   - 输入：自家产品描述 + 竞品名称/链接/粘贴的竞品文档。  
   - 输出 JSON：`{ "competitors": [...], "dimensions": [...], "comparison_table": [...], "swot_or_suggestions": "..." }`，或长 Markdown 报告。

2. **PRD 撰写**  
   - 输入：产品创意、目标用户、约束条件。  
   - 输出：结构化 PRD（目标、用户故事、功能列表、非功能需求、验收标准），可用 JSON + 一段 Markdown 正文。

3. **需求 PR 完善**  
   - 输入：原始需求或初版 PRD。  
   - 输出：补全后的 PRD（同结构），并标注“新增/修改”的段落或字段。

4. **需求分析（开发）**  
   - 输入：需求文档/PRD。  
   - 输出 JSON：功能拆解、技术影响模块、依赖、风险、排期建议。

5. **技术方案设计**  
   - 输入：需求/PRD + 可选“当前技术栈说明”。  
   - 输出：架构要点、接口设计、存储设计、关键流程、可选方案对比。

6. **需求分析（测试）**  
   - 输入：需求文档/PRD。  
   - 输出 JSON：测试视角的功能拆解、可测性、风险点、边界与异常建议。

7. **功能点梳理**  
   - 输入：需求文档/PRD。  
   - 输出 JSON：与用例智能体一致的 `modules[*].functions[*]`（可先不包含 cases），便于和现有用例生成链路打通。

---

## 八、与现有系统的衔接

- **需求管理**：`FunctionalRequirement` 已有需求名称、链接、负责人等，需求智能体生成的结果（如功能点、PRD）可写入需求备注或通过“关联文档”挂到该需求下（需后续扩展字段或新表）。
- **用例智能体**：测试侧“功能点梳理”输出可直接作为“需求输入”传给现有 `generate-stream`，实现「需求 → 功能点 → 用例」一条龙。
- **配置**：Kimi 继续用 `KIMI_*`；若需求智能体单独模型，可增加 `REQUIREMENT_AGENT_MODEL`、`REQUIREMENT_AGENT_BASE_URL` 等，与现有 env/settings 风格一致。

---

以上三种方案可根据“上线速度 vs 角色化 vs 多模型扩展”的优先级选择；推荐先落地方案一，再按需演进到方案三或方案二。
