# 需求智能体完整方案

本文整合「业界设计」「技术实现」「产品与开发使用」三部分，给出**需求智能体**的完整方案，便于评审、立项与实施。

---

## 第一部分：定位与目标

### 1.1 为什么做需求智能体

- **需求链路痛点**：产品从创意到 PRD 耗时长；开发消化需求、写技术方案重复劳动；测试理解需求、梳理功能点易漏；需求澄清分散在会议/聊天中，难以回写唯一真相源。
- **目标**：在**不替代人**的前提下，用 AI 在需求链路上做**初稿生成与辅助分析**，提升产品写 PRD、开发做分析与技术方案、测试做需求理解与功能点梳理的效率，并与现有**用例智能体**打通，形成「需求 → 功能点 → 用例」闭环。
- **与业界的关系**：对齐 AI-RSD（角色-任务-能力-评估）、Agentic PRD（机器可读、可验证验收标准）的演进方向；当前阶段以**单智能体多任务**落地，预留多智能体与按任务选模型的扩展空间。

### 1.2 范围与边界

| 纳入 | 不纳入（本期） |
|------|----------------|
| 产品：竞品分析、PRD 撰写、需求 PR 完善 | 自动创建 JIRA/Confluence、MCP 暴露给 IDE/CI |
| 开发：需求分析、技术方案设计 | 代码生成、自动排期 |
| 测试：需求分析（测试视角）、功能点梳理 | 自动执行测试、测试报告生成 |
| 与用例智能体的输入输出对接 | 多智能体协作（撰写+研究+评审）的完整 SOP |
| **多模态输入**：文字 + 文档（PDF/Word 等）+ 图片；**产品经理场景支持视频**（见 4.7） | 组织级 RAG/知识库、联网搜索 |

## 第二部分：业界对齐（设计依据）

### 2.1 需求规格的演进

- **传统 PRD 的局限**：意图模糊、行为非确定、缺乏机器可读结构、规格易腐化、歧义被 AI 放大。业界演进为 **AI-RSD**（角色-任务-能力-评估）与 **Agentic PRD**（声明式验收标准、示例、版本化）。
- **本方案**：输出结构上向「结构化 + 可追溯」靠拢（JSON + 可选 Markdown）；功能点梳理与用例智能体结构一致，便于下游自动化；定稿与唯一真相源仍由人在需求池/PRD 中维护。

### 2.2 智能体架构（四层）

| 层级 | 在本方案中的体现 |
|------|------------------|
| **感知** | **多模态**：文本 + 文档（PDF/Word 等）+ 图片，复用 FileProcessor；产品侧支持视频（关键帧/转写，见 4.7） |
| **推理** | 单模型按任务类型执行对应 Prompt，可选思考模式（use_thinking） |
| **记忆** | 短期：单次请求上下文；长期：预留扩展（历史任务结果、RAG） |
| **行动** | 生成结构化/文档输出并落库；与需求管理、用例智能体通过数据关联衔接 |

### 2.3 单智能体 vs 多智能体

- **本期**：采用**单智能体、多任务类型**（一个入口，按 `role` + `task_type` 选能力），实现简单、与现有用例智能体风格一致，适合先跑通全角色能力。
- **后续**：可演进为「按任务选模型」或「产品/开发/测试三个智能体入口」，以及「PRD 撰写 → 研究/评审 → 修订」的多智能体 SOP，按质量与成本需求再投入。

## 第三部分：能力与范围

### 3.1 能力矩阵

| 角色 | 任务类型（task_type） | 输入 | 输出形态 |
|------|------------------------|------|----------|
| **产品** | competitive_analysis | 自家产品描述 + 竞品名/链接/文档/**图片/视频**（典型：竞品演示视频、宣传片、录屏直接上传） | 结构化报告（维度对比、优劣势、建议）或 Markdown |
| **产品** | prd_draft | 创意、目标用户、约束/**图片/视频/文档** | PRD 文档（目标、用户故事、功能列表、非功能需求、验收标准） |
| **产品** | prd_refine | 原始需求/初版 PRD + 澄清点/**截图/录屏** | 完善后的 PRD，可标注新增/修改 |
| **开发** | requirement_analysis | 需求文档/PRD | 功能拆解、技术影响面、风险与依赖、排期建议（JSON） |
| **开发** | tech_design | 需求/PRD + 可选技术栈说明 | 技术方案（架构、接口、存储、关键流程）（JSON + Markdown） |
| **测试** | test_requirement_analysis | 需求文档/PRD | 测试视角分析：可测性、风险点、边界与异常（JSON） |
| **测试** | feature_breakdown | 需求文档/PRD | 功能点清单（模块/功能点/验收要点），与用例智能体结构一致（JSON） |

### 3.2 角色与使用场景小结

| 角色 | 主要使用场景 | 输入从哪来 | 输出给谁用 |
|------|--------------|------------|------------|
| 产品 | 立项前竞品分析、从创意到 PRD、PRD 完善与澄清 | 创意/会议结论/竞品链接/初版 PRD | 开发、测试、需求池 |
| 开发 | 需求评审前分析、技术方案设计 | 产品 PRD/需求文档 | 排期、设计评审、测试（可选） |
| 测试 | 需求理解、功能点梳理、为用例生成做准备 | 产品 PRD/需求文档、可选技术方案 | 用例智能体、测试计划 |

## 第四部分：技术方案

### 4.1 推荐实现：单应用多任务类型

- **应用**：新建 `apps.ai_requirement`，与现有 `apps.ai_testcase` 并列。
- **入口**：统一 `POST /api/ai_requirement/run-stream/`，Body 含 `role`、`task_type`、`requirement`、`files`（可选）、`use_thinking`（可选）。
- **复用**：现有 Kimi 客户端（或抽象为通用 LLM 客户端）、FileProcessor（上传文件提取文本/图片）、SSE 流式、配置（KIMI_*）；产品经理场景扩展**视频**支持（见 4.7）。
- **扩展**：后续可增加「按 task_type 选模型」配置、或拆出 product/dev/test 三个子路径，底层仍共用 Prompt 与数据模型。

### 4.2 模型选型

- **主模型**：与用例智能体一致，**Kimi K2.5**（Moonshot API），支持长上下文、思考模式、多模态。
- **配置**：使用现有 `KIMI_API_KEY`、`KIMI_BASE_URL`、`KIMI_MODEL`；若需求智能体需独立模型，可增加 `REQUIREMENT_AGENT_MODEL`（可选）。
- **后续**：可为不同 task_type 配置不同模型（如功能点梳理用更轻量模型），通过配置表或环境变量映射。

### 4.3 数据模型

```text
AiRequirementTask
├── role (product | dev | test)
├── task_type (competitive_analysis | prd_draft | prd_refine | requirement_analysis | tech_design | test_requirement_analysis | feature_breakdown)
├── requirement_input (TextField)          # 用户输入的需求/PRD/描述
├── status (pending | generating | success | failed)
├── result_json (JSONField, nullable)     # 结构化结果
├── result_md (TextField, nullable)      # 长文档结果（PRD/报告等）
├── source_files (JSONField, nullable)   # 上传文件路径列表，同 ai_testcase
├── prompt_tokens, completion_tokens, total_tokens
├── error_message (TextField, nullable)
├── created_at, updated_at
└── 可选：requirement_id (FK to FunctionalRequirement)，便于与需求管理关联
```

### 4.4 接口约定

- **流式**：SSE，事件类型与现有用例智能体一致：`start`（含 record_id、附件信息等）→ `chunk`（流式内容）→ `done`（含 data / result_md、usage）或 `error`。
- **非流式**（可选）：`POST /api/ai_requirement/run/` 返回完整结果，便于集成与调试。

### 4.5 输出结构约定（与现有体系对齐）

- **功能点梳理（feature_breakdown）**：输出与 `ai_testcase.result_json` 的模块/功能点部分一致，即 `{ "title": "...", "modules": [ { "name": "...", "functions": [ { "name": "..." } ] } ] }`，不包含 cases，便于直接作为用例智能体的「需求+结构」输入。
- **需求分析（开发/测试）**：统一为 JSON，如 `{ "summary": "...", "breakdown": [...], "risks": [...], "dependencies": [...] }`，便于前端展示与后续消费。
- **竞品分析**：JSON（维度、对比表、建议）或长 Markdown 存 `result_md`。
- **PRD 撰写/完善**：结构化字段可放 `result_json`，正文放 `result_md`；或整份 PRD 以 Markdown 存 `result_md`。

### 4.6 Prompt 设计要点

- 每个 (role, task_type) 对应一套 **system prompt** + **user 组装规则**（把 requirement、附件文本/图片拼进 user）。
- **原则**：角色化（产品经理/架构师/测试分析师）、输出格式明确（JSON 字段或 Markdown 结构）、禁止幻觉（缺失信息时标注「需补充」而非编造）。
- 多模态：若存在 `files`，复用 ai_testcase 的「多模态 prompt」模式（文本 + 附件摘要 + 图片 base64）；产品侧若含视频，采用 4.7 节方案后一并拼入 prompt。

### 4.7 多模态支持（产品经理：视频、图片、文本）

需求智能体**支持多模态**，产品经理使用时需要**视频、图片、文本**同时作为输入（如：竞品录屏 + 原型截图 + 需求描述）。

**典型场景：竞品调研时上传竞品视频**  
产品在做竞品分析时，常会把**竞品的产品演示视频、宣传片、功能录屏**等直接丢给智能体，例如：竞品 App 的录屏、官网宣传视频、发布会片段。智能体需要能「看懂」这些视频内容（界面、流程、卖点），并结合文本描述输出竞品对比或 PRD 参考。因此**视频支持是竞品分析（competitive_analysis）场景的强需求**，二期实现视频能力时，可优先在「竞品分析」任务中开放上传视频，再推广到 PRD 撰写、需求完善等。

| 模态 | 当前支持 | 说明 |
|------|----------|------|
| **文本** | ✅ | 输入框 + 文档（PDF/Word/Txt/Md）提取正文，沿用 FileProcessor |
| **图片** | ✅ | 独立图片（png/jpg/webp）+ 文档内嵌图，提取后以 base64 入 vision API |
| **文档** | ✅ | PDF/Word 等提取文字 + 内嵌图，同上 |
| **视频** | 方案支持（需扩展） | 见下方「视频支持方案」 |

**视频支持方案**（产品侧必选）：

- **方案 A：关键帧抽取（推荐）**  
  - 上传视频（如 mp4/webm），服务端按间隔或场景抽取关键帧（如每 3～5 秒一帧或基于镜头切换），将关键帧转为图片后送入现有**多模态**流程（与图片一致）。  
  - 优点：复用现有 vision 能力，无需模型原生支持视频；实现简单（ffmpeg/OpenCV 抽帧 + 现有图片压缩与 base64）。  
  - 缺点：无语音信息，若需「会议录屏」内容，需配合方案 B。

- **方案 B：视频转文字（ASR）+ 关键帧**  
  - 对视频做语音识别（ASR）得到字幕/文稿，再抽少量关键帧作为配图；prompt 中拼入「视频转写文字」+「以下为视频关键帧截图」。  
  - 优点：录屏、会议回放等带语音场景信息完整。  
  - 缺点：需接入 ASR 服务（如阿里/腾讯/讯飞或开源 Whisper），成本与依赖增加。

- **方案 C：模型原生视频（若可用）**  
  - 若所选 API（如后续 Kimi 或其它供应商）支持直接输入视频 URL 或视频 token，可优先采用，无需自建抽帧与 ASR。  
  - 当前需确认 Kimi/Moonshot 是否支持 video 输入；若不支持，本期以方案 A 或 A+B 实现。

**实施建议**：

1. **一期**：产品侧多模态 = **文本 + 图片 + 文档**（与现有 FileProcessor 一致），先不接视频。  
2. **二期**：增加视频上传与**关键帧抽取**（方案 A），关键帧与现有图片同一路径送入多模态 Prompt；**优先支持「竞品分析」任务**——产品经理可把竞品视频（演示录屏、宣传片、功能录屏）丢给智能体做分析，再推广到 PRD 撰写、需求完善等场景。  
3. **可选**：若产品强需求「会议/录屏中的语音」，再接入 ASR（方案 B），将转写文本与关键帧一并送入。

**前端与接口**：

- 产品经理工作台支持：输入框（文本）+ 上传（文档 + 图片 + **视频**）。  
- 视频格式建议：mp4、webm；单文件大小与时长限制需约定（如 ≤50MB、≤5 分钟），避免 token/成本过高。  
- 接口 `run-stream` 的 `files` 中允许视频文件；服务端根据扩展名识别视频，走视频处理管道（抽帧/ASR），结果与现有 texts/images 一起组装进多模态消息。

### 4.8 与现有系统衔接

- **需求管理**：`FunctionalRequirement` 可扩展「关联 AI 任务」或「关联文档」；需求智能体生成的功能点/PRD 片段可写入备注或通过关联表挂到需求下，便于追溯。
- **用例智能体**：测试侧「功能点梳理」结果 + 需求摘要，作为 `ai_testcase` 的 `generate-stream` 输入，实现「需求 → 功能点 → 用例」一条龙；前端可提供「从需求智能体结果一键生成用例」入口。
- **配置**：与现有 env/settings 一致，Kimi 共用；若单独模型则增加 REQUIREMENT_AGENT_* 配置项。

## 第五部分：使用方案

### 5.1 在现有研发流程中的接入点

```text
[产品] 创意/会议结论
    → 竞品分析（可选）→ PRD 撰写 / 需求 PR 完善 → PRD 定稿进入需求池

[开发] 拿到 PRD
    → 需求分析 → 需求评审（人工）→ 技术方案设计 → 排期与设计评审

[测试] 拿到 PRD（及可选技术方案）
    → 需求分析（测试视角）→ 功能点梳理 → 用例智能体生成用例 → 测试计划/用例评审
```

- 产品：写 PRD、完善需求时使用；竞品分析在立项/方案前使用。
- 开发：评审前做需求分析、设计前做技术方案；输入为产品 PRD。
- 测试：理解需求与写用例前做需求分析 + 功能点梳理；功能点输出直接对接用例智能体。

### 5.2 三种使用方式

| 方式 | 做法 | 适用 |
|------|------|------|
| **单次使用** | 选角色+任务类型，输入材料，拿结果后人工修改并入文档 | 日常零散、试点 |
| **流水线使用** | 多步串联（如竞品→PRD 初稿→完善），每步可编辑再进入下一步 | 希望一气呵成 |
| **多智能体协作** | 撰写→研究/评审→修订多角色依次执行（进阶） | 质量要求高、后续迭代 |

### 5.3 上下游衔接

- **产品 → 开发/测试**：同一份 PRD 定稿作为需求分析、技术方案、需求分析（测试）、功能点梳理的输入；需求管理页可关联 PRD 链接或粘贴内容调用智能体。
- **开发 → 测试（可选）**：技术方案可作为测试「需求分析」或用例设计的补充输入，便于覆盖接口与异常场景。
- **测试内部**：功能点梳理结果作为用例智能体的输入，形成闭环；测试计划中可记录「功能点来源」与「用例生成记录」。

### 5.4 使用规范（人机协同）

- 智能体产出一律视为**初稿**，须经责任人（产品/开发/测试）校验与定稿。
- 以「需求池/PRD 定稿」为**唯一真相源**；智能体只生成草稿，定稿由人更新到正式文档或需求系统。
- 敏感需求可限制上传内容或仅用脱敏摘要；审计日志记录谁在何时调用了哪类任务（若具备权限与审计需求）。

## 第六部分：落地与实施

### 6.1 实施阶段

| 阶段 | 内容 |
|------|------|
| **阶段 0：准备** | 确定入口（需求管理页/独立工作台）、输入输出模板（含功能点与用例智能体对接格式）、权限与审计策略 |
| **阶段 1：单点试点** | 产品先开 PRD 撰写或需求完善；开发先开需求分析或技术方案；测试打通「功能点梳理 → 用例智能体」；收集反馈、打磨 Prompt |
| **阶段 2：按角色推广** | 产品纳入竞品+PRD+完善；开发评审前需求分析、设计前技术方案；测试功能点+用例智能体为标配；在需求管理/Wiki 留使用指引 |
| **阶段 3：固化与度量** | 在需求/项目配置中固化「建议使用节点」；度量使用次数/角色、澄清次数、缺陷逃逸等，持续改进 |

### 6.2 开发实施顺序（建议）

1. 新建 `apps.ai_requirement`，定义 `AiRequirementTask` 模型及迁移。
2. 实现统一 `run-stream` 接口，路由 `role`+`task_type` 到对应 Prompt 与输出解析；复用 Kimi 客户端与 FileProcessor。
3. 按任务类型实现 7 类 Prompt（竞品分析、PRD 撰写、PRD 完善、需求分析、技术方案、测试需求分析、功能点梳理）及结果解析/落库。
4. 前端：需求管理页或独立「需求智能体」页，支持选择角色/任务、输入/上传、流式展示结果、保存/关联到需求。
5. 打通「功能点梳理 → 用例智能体」：功能点结果可一键作为用例生成输入（含需求摘要）。
6. （可选）按任务选模型、产品/开发/测试分入口、与 FunctionalRequirement 关联字段与列表展示。

### 6.3 风险与治理

| 风险 | 应对 |
|------|------|
| 直接当终稿用 | 规范：所有产出为初稿，必须人工校验与签字 |
| 需求不一致 | 唯一真相源为需求池/PRD 定稿；智能体只读或只出草稿 |
| 过度依赖 | 定期保留「不用智能体」的演练或评审，保持人的判断力 |
| 隐私与安全 | 敏感需求限制上传或脱敏；审计日志记录调用方与任务类型 |

## 第七部分：附录

### 7.1 任务类型枚举（task_type）

| 值 | 说明 | 角色 |
|----|------|------|
| competitive_analysis | 竞品分析 | product |
| prd_draft | PRD 撰写 | product |
| prd_refine | 需求 PR 完善 | product |
| requirement_analysis | 需求分析 | dev |
| tech_design | 技术方案设计 | dev |
| test_requirement_analysis | 需求分析（测试视角） | test |
| feature_breakdown | 功能点梳理 | test |

### 7.2 功能点梳理输出结构（与用例智能体对齐）

```json
{
  "title": "需求名称_功能点",
  "modules": [
    {
      "name": "模块名称",
      "functions": [
        { "name": "功能点名称" }
      ]
    }
  ]
}
```

该结构可直接与 `ai_testcase` 的 `result_json` 的 modules/functions 部分兼容；传入用例智能体时，可据此生成 cases，或由用户补充需求描述后生成完整用例。

**完整方案小结**：需求智能体以**单应用多任务类型**实现产品（竞品分析、PRD 撰写、需求完善）、开发（需求分析、技术方案）、测试（需求分析、功能点梳理）七类能力；复用现有 Kimi、FileProcessor、SSE；与需求管理、用例智能体通过数据与入口衔接；使用上遵循「谁负责谁发起、下游复用上游、人工定稿为唯一真相源」；落地按准备→单点试点→按角色推广→固化与度量推进，并预留按任务选模型与多智能体扩展空间。
