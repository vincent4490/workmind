# 业界需求智能体设计综述

本文从**业界实践与学术/工业框架**出发，综述需求类智能体（含 PRD、需求分析、竞品分析、技术方案等）的常见设计方式，不依赖具体项目实现。

---

## 一、设计范式：从 PRD 到 AI-RSD / Agentic PRD

### 1.1 传统 PRD 在 AI 时代的失灵

业界普遍认为，面向「人传人」的 PRD 在 AI 参与研发时暴露出明显局限：

| 问题 | 说明 |
|------|------|
| **意图模糊性** | 难以描述深层、隐含的用户意图，AI 需要可推演的目标而非操作步骤 |
| **行为非确定性** | AI 输出具有概率性，无法穷举和用传统验收标准覆盖 |
| **能力动态性** | 模型迭代快，静态文档跟不上智能体能力演进 |
| **缺乏机器可读结构** | 自然语言段落无法被 IDE、CI、测试生成器直接消费 |
| **规格腐化** | 澄清发生在会议/聊天中，很少回写到唯一真相源（PRD） |
| **歧义被放大** | 需求模糊时，模型会自行“发明”默认行为，导致偏差 |

因此业界出现两条演进路径：**AI-RSD（需求规格设计）** 与 **Agentic PRD（面向智能体的 PRD）**。

### 1.2 AI-RSD：角色-任务-能力-评估

**AI-RSD** 是一套为智能体设计需求规格的框架，核心是四要素：

1. **角色**：智能体的身份、语气、价值观与能力边界  
2. **任务**：要解决的**问题与用户意图**，而非一步步操作说明  
3. **能力**：定义「可能性边界」（能调用哪些工具、能做哪些规划与推理），而非精确输出  
4. **评估**：对结果、过程、可靠性、安全建立可观测指标，形成闭环  

思维转变：

- 从「精确功能描述」→「行为概率与价值对齐」  
- 从「用户做什么」→「用户想达成什么」  
- 从「静态规约」→「系统性风险管理」

### 1.3 Agentic PRD：机器可读、可执行的需求

**Agentic PRD** 强调需求文档同时面向**人与机器**，具备：

- **声明式、可验证的验收标准**：原子谓词或结构化断言，而非纯自然语言列表  
- **输入/输出示例**：作为 ground truth，减少幻觉、支撑测试生成  
- **非功能需求一等公民**：延迟、安全等以可校验形式存在（如 `max_ms: 300`）  
- **语义元数据与术语表**：领域词定义，减少歧义  
- **版本化、不可变快照**：PR/提交可引用 PRD 版本，CI 可校验一致性  

典型工作流：PM 在平台编写结构化 PRD → 发布规范版本 → IDE/CI 通过 MCP 等协议拉取验收条件与示例 → 自动生成测试脚手架、Mock、策略门禁。

## 二、智能体核心架构：四层模型

业界常用「四层」来刻画单个智能体：

| 层级 | 职能 | 常见实现 |
|------|------|----------|
| **感知层** | 理解多模态输入与上下文 | 文本/语音/图像理解、意图识别、上下文窗口 |
| **推理层** | 规划、决策、任务分解 | LLM 驱动的推理与规划（含 ReAct、思维链等） |
| **记忆层** | 短期与长期知识 | 短期：对话/上下文窗口；长期：向量库、RAG、知识库 |
| **行动层** | 与外界交互 | 工具调用（API、数据库、搜索、代码执行）、写回文档/工单 |

需求智能体通常会：

- **感知**：需求描述、上传的 PRD/竞品文档、会议纪要、链接等  
- **推理**：拆解为竞品分析 / PRD 撰写 / 需求完善 / 技术方案等子任务  
- **记忆**：历史需求、组织术语表、既有 PRD 片段（RAG）  
- **行动**：生成文档、调用搜索/知识库、创建 JIRA/工单、触发下游流水线

## 三、单智能体 vs 多智能体

### 3.1 单智能体

- **适用**：任务边界清晰、步骤可预测、一种角色即可完成。  
- **实现**：一个 LLM + 一套（或按任务类型分支的）Prompt + 工具集，流式或非流式输出。  
- **优点**：简单、延迟低、成本可控。  
- **缺点**：复杂需求（如「调研+撰写+评审+修订」）容易一次承担过多，质量与一致性依赖单一模型上限。

### 3.2 多智能体协作（业界主流用于 PRD/需求）

**思路**：把「写 PRD / 做需求分析」拆成多角色、多步骤，由不同智能体分工协作，互相评审与修订。

**MetaGPT 类设计（以 PRD 生成为例）**：

- **环境**：共享消息池（全局上下文），智能体通过消息交换信息，而非直接调用。  
- **角色**：例如  
  - **Product Manager（协调者）**：起草初版 PRD、收集研究/评审反馈、修订并定稿；  
  - **Researcher**：做市场/竞品调研，补充数据与依据；  
  - **Reviewer**：评审 PRD，给出可执行的改进建议。  
- **流程（SOP）**：  
  1. 用户输入创意/需求概要  
  2. PM 生成 DraftPRD.md（目标、用户画像、功能要点）  
  3. Researcher 基于草稿做研究并输出补充内容  
  4. Reviewer 对草稿做评审并输出反馈  
  5. PM 汇总研究与评审，修订并输出最终 PRD  
  6. 将结果写入版本化存储（如 PRD.md）  

每个智能体内部是「观察 → 思考 → 行动 → 共享」的循环，直到满足终止条件（如轮数或质量门禁）。

**MARE 框架（需求工程全流程）**：

- **四类任务**：Elicitation（ elicitation）、Modeling（建模）、Verification（验证）、Specification（规格编写）。  
- **五类智能体 + 九种动作**：不同智能体负责不同任务，在**共享工作区**上传/读取中间产物（需求草稿、模型、规格片段）。  
- **效果**：在部分实验中相较基线在「正确需求模型」上提升约 15.4%，且支持人类参与（human-in-the-loop）。

**Elicitron（需求 elicitation）**：

- 用多智能体**模拟多样用户**，替代或补充传统用户访谈。  
- 不同 Agent 扮演不同用户视角，挖掘潜在与隐含需求。  
- 研究表明：上下文感知的 Agent 能产生更丰富的需求多样性，并识别出更多潜在需求。

**共同点**：

- 角色专精（PM / 研究员 / 评审 / 用户模拟等）  
- 共享环境或工作区  
- 明确 SOP 或工作流阶段  
- 中间产物可被下游智能体读取与迭代  

### 3.3 何时用单智能体、何时用多智能体

- **单智能体**：  
  - 单一明确任务（如「根据这份 PRD 做功能点梳理」「根据摘要写一段竞品对比」）。  
  - 对延迟和成本敏感，且质量要求可在单模型内满足。  
- **多智能体**：  
  - 需要「调研 + 撰写 + 评审 + 修订」完整链路。  
  - 需要不同专业视角（产品、研发、测试、用户）共同参与。  
  - 对质量与一致性要求高，愿意用更多轮次和成本换取更好结果。

## 四、工作流与编排

### 4.1 流水线式（DAG）

- 节点：信息收集 → 市场/竞品分析 → PRD 撰写 → 完善/修订。  
- 每步可由不同模型或不同 Prompt 完成，可插入人工审核节点。  
- 平台示例：腾讯元器等工作流平台，用「信息收集节点 + 大模型节点 + 搜索/知识库节点 + 循环/判断节点」搭建 PRD 智能体；关键点：**只有信息收集节点可暂停等待用户输入**，其余为自动处理。

### 4.2 规划-执行图（Planner-Executor）

- **Planner**：根据用户目标生成计划（先竞品分析再写 PRD 再评审）。  
- **Executor**：按步骤调用专门的能力模块（Skill）执行。  
- 开源示例：product-agents 等仓库采用「前端 → 薄 API → 编排层 → 多个 Skill 模块」的 TypeScript 架构，适合 PRD 生成与结构化输出。

### 4.3 自主反射与多轮质检

- 智能体对自身输出做多轮检查（如一致性、覆盖度、是否符合模板）。  
- 未通过则自动修订或标注「需人工澄清」，形成「生成 → 自检 → 修订」闭环。  
- 企业实践中有「多评委验证」（多算法：自适应共识、原子声明分解等）+ 交互式澄清，使置信度达到 95%+ 再产出最终 PRD、JIRA、测试用例与验证报告。

## 五、模型选型与工具

### 5.1 模型角色

- **主推理/撰写**：长上下文、强推理模型（如 DeepSeek-R1、GPT-4o、Kimi K2.5）用于 PRD 撰写、技术方案、竞品分析长文。  
- **专项任务**：用户故事生成、优先级评估等，有研究用 GPT-3.5/4、LLaMA3-70、Mixtral-8B 等对比，结论是不同任务上优劣不同（如 Mixtral-8B 响应快，GPT-3.5 在复杂用户故事上更好）。  
- **本地/隐私**：Ollama + DeepSeek 等本地部署，适合无 API、离线或数据不出域的场景。

### 5.2 工具增强（Tool-augmented）

- **搜索**：竞品、市场数据、技术资料。  
- **知识库/RAG**：组织内部 PRD 模板、历史需求、术语表。  
- **下游系统**：JIRA/工单创建、Confluence/Notion 写回、IDE/CI 通过 MCP 读取验收标准与示例。  

需求智能体不应只是「一次生成一段话」，而应能**拉取上下文**（RAG/搜索）和**写回结构化结果**（工单、PRD 版本、测试用例草稿）。

## 六、质量与可观测性

### 6.1 评估维度（对齐 AI-RSD）

- **结果**：任务完成率、输出质量（人工或模型评分）、满意度、延迟。  
- **过程**：工具调用是否合理、规划是否可解释、错误恢复率。  
- **可靠性**：一致性、鲁棒性（多次运行或扰动下的表现）。  
- **安全与合规**：权限、审计、伦理与合规策略。

### 6.2 人机协同

- **置信度**：根据示例完整度、语义匹配度等给出置信度；低置信度时要求 PM/需求方确认后再写入规范或生成测试。  
- **需澄清标记**：当验收标准缺少谓词或示例时，标记 `requires-clarification`，阻塞自动测试生成直至澄清。  
- **门禁**：CI 中检查 PRD 版本、PR 元数据、PRD 衍生测试是否存在且通过，再允许合入。

## 七、业界小结：需求智能体设计要点

1. **需求规格本身要升级**：从「人读 PRD」到「人机共读」的 Agentic PRD / AI-RSD，强调角色-任务-能力-评估、可验证验收条件、示例与版本化。  
2. **架构**：感知-推理-记忆-行动四层；复杂需求场景多用**多智能体分工**（PM + Researcher + Reviewer 等）+ **共享环境/SOP**。  
3. **工作流**：流水线（DAG）或规划-执行图；关键节点可插入人工审核与置信度门禁。  
4. **工具与集成**：搜索、RAG、写回工单/文档、通过 MCP 等暴露给 IDE/CI，使需求与代码/测试可追溯。  
5. **质量**：多评委/多轮自检、置信度评分、人机协同澄清；CI 门禁与 PRD 版本绑定，形成闭环。  
6. **模型**：按任务选型（长文/推理用大模型，简单结构化任务可用小模型）；可混合云 API 与本地部署以平衡成本与隐私。

若把「需求智能体」定义为：**在需求 elicitation、分析、PRD 撰写与完善、竞品分析、技术方案、测试视角分析等环节中，能感知输入、推理规划、调用工具、产出结构化或可执行产物的系统**，则业界共识是：**单任务用单智能体+好 Prompt+工具即可；完整链路、高质量 PRD 与多角色视角则更适合多智能体 + 明确 SOP + 人机协同**。
