# 需求智能体 → 用例智能体桥接说明

本文档说明「功能点梳理」结果如何与「AI 用例智能体」对接的两种方式（方案 A 结构直传、方案 B 原文跳转），以及接口与数据流细节。

---

## 一、方案概览

| 方式 | 触发条件 | 数据流 | 用户体验 |
|------|----------|--------|----------|
| **方案 A（结构直传）** | 桥接接口返回了 `structure` 且 `structure.modules` 非空 | 需求侧 → 桥接 → 前端拿到 structure → 直接调用例侧 `generate-from-structure` → 只生成 cases → 跳转用例页并打开该记录 | 一键完成：点「生成测试用例」后自动生成并打开结果，无需在用例页再点「智能生成」 |
| **方案 B（原文跳转）** | 桥接未带 `structure`，或仅老版本前端 | 需求侧 → 桥接 → 前端拿到 `requirement_text` → 存 sessionStorage → 跳转用例页 → 用例页预填需求框，用户点击「智能生成用例」调用现有 `generate-stream` | 与改造前一致：先跳转再在用例页用「需求描述」生成 |

两套逻辑并存：**方案 A 不替代方案 B**，前端根据桥接返回是否含 `structure` 自动选择走哪条路。

---

## 二、方案 A（结构直传）详细说明

### 2.1 流程概览

```
[需求页] 用户完成「功能点梳理」任务
    → 点击「生成测试用例」
    → 前端调用 POST /api/ai_requirement/bridge-to-testcase/ { task_id }
    → 后端返回 { status, requirement_text, structure, source_task_id, module_count, function_count }
    → 若存在 structure 且 structure.modules.length > 0：
        → 前端调用 POST /api/ai_testcase/generate-from-structure/（SSE）
          请求体：{ structure, source_requirement_task_id?, requirement?, use_thinking? }
        → 后端按「模块 / 功能点 / 验收点」只生成 cases，流式返回
        → 前端 onDone 收到 record_id
        → 跳转至用例页并带 query：?record_id=xxx
    → 用例页 onMounted 发现 record_id，请求预览该记录并展示，然后去掉 query
```

### 2.2 桥接接口（需求侧）

- **URL**：`POST /api/ai_requirement/bridge-to-testcase/`
- **请求体**：`{ "task_id": 123 }`
- **约束**：仅支持 `task_type === 'feature_breakdown'` 且任务有 `result_json`。

**响应示例**：

```json
{
  "status": "ok",
  "requirement_text": "# 功能点梳理\n\n## 模块：用户管理\n\n### 登录（P0）\n描述：...\n  - 验收点1\n  - 验收点2",
  "structure": {
    "title": "功能点梳理",
    "modules": [
      {
        "name": "用户管理",
        "functions": [
          {
            "name": "登录",
            "description": "...",
            "acceptance_points": ["验收点1", "验收点2"],
            "priority": "P0",
            "test_hint": "建议覆盖异常登录"
          }
        ]
      }
    ]
  },
  "source_task_id": 123,
  "module_count": 1,
  "function_count": 1
}
```

- **requirement_text**：把功能点拼成 Markdown 文本，供方案 B 或人工阅读。
- **structure**：与 feature_breakdown 的 `result_json` 同构，但每个 function 上**已去掉 `cases`**，由用例接口负责生成。前端拿到后直接传给 `generate-from-structure`，无需再解析。

实现位置：`backend/apps/ai_requirement/views.py` 中 `bridge_to_testcase_view`（构造 `structure` 时对 `fb_data` 深拷贝并 `func.pop('cases', None)`）。

### 2.3 用例生成接口（用例侧，方案 A 专用）

- **URL**：`POST /api/ai_testcase/generate-from-structure/`
- **Content-Type**：`application/json`
- **请求体**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| structure | object | 是 | 与 feature_breakdown 输出一致：`title`、`modules[].name`、`modules[].functions[]` 含 `name`、`description`、`acceptance_points`、`priority`、`test_hint` 等；**不要带 `cases`**，由本接口生成 |
| requirement | string | 否 | 需求摘要，写入生成记录的 `requirement` 字段，并作为 prompt 补充说明 |
| use_thinking | boolean | 否 | 默认 false，是否开启深度推理 |
| source_requirement_task_id | number | 否 | 需求侧任务 ID，用于前端/后续血缘回写，SSE 的 start 事件会回带 |

- **响应**：`text/event-stream`，与现有 `generate-stream` 事件格式一致。

**SSE 事件**：

- `data: {"type":"start","record_id":1,"source_requirement_task_id":123}\n\n`
- `data: {"type":"chunk","content":"..."}\n\n`
- `data: {"type":"done","record_id":1,"title":"...","module_count":n,"case_count":m,"usage":{...},"data":{...}}\n\n`
- `data: {"type":"error","error":"...","record_id":1}\n\n`

后端逻辑要点：

- 只根据「模块 / 功能点 / 验收点」生成 **cases**，不重新从自然语言理解需求。
- Prompt 见 `backend/apps/ai_testcase/services/prompts.py` 中 `TESTCASE_FROM_STRUCTURE_SYSTEM_PROMPT` 与 `get_testcase_from_structure_prompt`。
- 流式生成、解析 JSON、落库、生成 XMind、返回事件与 `generate-stream` 一致。实现：`views.py` 中 `generate_from_structure_stream_view`，`kimi_client.py` 中 `generate_testcases_from_structure_stream_async`。

### 2.4 前端方案 A 行为

- **需求页**（`AiRequirementAgent.vue`）：  
  `handleBridgeToTestcase` 里若 `res.structure && res.structure.modules?.length`，则调用 `aiGenerateTestcaseFromStructureStream({ structure: res.structure, source_requirement_task_id: res.source_task_id, use_thinking: false }, ...)`，在 `onDone` 中拿到 `event.record_id`，执行 `router.push({ name: 'AiTestcaseGenerator', query: { record_id: event.record_id } })`。
- **用例页**（`AiTestcaseGenerator.vue`）：  
  `onMounted` 中若存在 `route.query.record_id`，先 `loadHistory()`，再 `previewAiTestcase(record_id)` 设置 `currentResult` 并展示，最后 `router.replace({ name: 'AiTestcaseGenerator' })` 去掉 query，避免刷新重复打开。

## 三、方案 B（原文跳转）详细说明

### 3.1 流程概览

```
[需求页] 用户完成「功能点梳理」
    → 点击「生成测试用例」
    → 前端调用 bridge-to-testcase
    → 若返回无 structure（或前端不处理 structure）：
        → 将 res.requirement_text 存入 sessionStorage['ai_req_bridge_text']
        → 将 res.source_task_id 存入 sessionStorage['ai_req_bridge_source']
        → 跳转至用例页（无 query）
[用例页] onMounted
    → 读取 ai_req_bridge_text，填入「需求描述」输入框
    → 提示「已从 AI 需求智能体导入功能点，可直接点击智能生成用例」
    → 用户点击「智能生成用例」→ 调用现有 POST /api/ai_testcase/generate-stream/（requirement + 可选文件）
    → 后端按「需求描述」全文理解并生成 模块/功能点/用例 完整树
```

### 3.2 与方案 A 的差异

- **输入**：方案 B 的输入是「需求描述」长文本（由桥接把功能点拼成 Markdown）；方案 A 的输入是结构化 JSON（`structure`）。
- **用例侧接口**：方案 B 使用 **generate-stream**（从自然语言生成完整用例树）；方案 A 使用 **generate-from-structure**（只补全 cases）。
- **体验**：方案 B 需要用户在用例页再点一次「智能生成用例」；方案 A 在需求页点一次即可完成生成并自动打开结果。

现有 **generate-stream** 与 **sessionStorage** 用法均保留，未做破坏性修改。

---

## 四、兼容性与保留能力

- **桥接接口**：始终返回 `requirement_text` 与 `source_task_id`；在支持 feature_breakdown 且 `result_json` 含 `modules` 时**额外**返回 `structure`。老前端只读 `requirement_text` 仍可走方案 B。
- **用例页**：  
  - 有 `record_id` query → 打开该次生成记录（方案 A 跳转）。  
  - 有 `ai_req_bridge_text` → 预填需求框（方案 B）。  
  两种可并存，优先处理 `record_id`。
- **generate-stream**：仍用于「需求描述 + 可选附件」的完整生成，与 generate-from-structure 并行存在，互不影响。

---

## 五、回写 downstream_testcases（已实现）

**含义**：用例智能体在方案 A 下生成完成后，将本次生成的用例记录 ID 写回需求任务，便于「需求任务 → 下游用例」血缘追踪。

- **数据模型**：`AiRequirementTask.downstream_testcases` 为 JSON 数组，存下游用例生成记录 ID 列表（如 `[1, 2, 3]`）。见 `backend/apps/ai_requirement/models.py`。
- **当前实现（后端回写）**：在 `generate-from-structure` 的 **done** 分支中，若请求体带了 `source_requirement_task_id`，则在保存完 `AiTestcaseGeneration` 后，查 `AiRequirementTask` 并将本次 `record_id` **追加**到 `downstream_testcases` 再 save；若任务不存在则打 warning 日志，不抛错。实现位置：`backend/apps/ai_testcase/views.py` 中 `generate_from_structure_stream_view` 的 event_stream 内。
- **去重**：若该 `record_id` 已在列表中则不再重复追加。

## 六、接口速查

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/ai_requirement/bridge-to-testcase/` | POST | 桥接：入参 `task_id`，返回 `requirement_text`、`structure`（可选）、`source_task_id` 等 |
| `/api/ai_testcase/generate-from-structure/` | POST | 方案 A：入参 `structure` 等，SSE 流式返回，只生成 cases |
| `/api/ai_testcase/generate-stream/` | POST | 方案 B / 独立使用：入参 `requirement`（+ 可选文件），SSE 流式返回，生成完整用例树 |
| `/api/ai_testcase/generations/{id}/preview/` | GET | 用例页根据 `record_id` 打开某次生成结果 |

以上为「需求智能体与用例智能体桥接」的详细说明；若需再补「回写 downstream_testcases」或接口文档的细粒度改动说明，可在此基础上追加一节即可。
