# 流式实现对比：用例生成 vs 需求智能体

## 一、后端对比

### 1. 用例生成（ai_testcase）

**入口**：`POST /api/ai_testcase/generate-stream/`  
**视图**：`backend/apps/ai_testcase/views.py` → `generate_stream_view`

| 项目 | 实现 |
|------|------|
| 流式来源 | `KimiClient.generate_testcases_stream_async()` / `generate_testcases_multimodal_stream_async()`，内部 `async for chunk in stream` 逐 token 从 Kimi 拉取 |
| event_stream | `async def event_stream()`，`async for event in gen` 中 `event['type'] == 'chunk'` 时直接 `yield f"data: {json.dumps({'type': 'chunk', 'content': event['content']}, ...)}\n\n"` |
| yield 后让出 | **无** `await asyncio.sleep(0)` |
| 响应头 | `Cache-Control: no-cache`、`X-Accel-Buffering: no`、**`Content-Encoding: identity`**（注释：防止 GZipMiddleware 压缩 SSE 流，压缩会导致缓冲） |
| 其他 SSE 流 | `regenerate_module_stream_view`、`review_stream_view`、`apply_review_stream_view` 等同样设置 `Content-Encoding: identity` |

### 2. 需求智能体（ai_requirement）

**入口**：`POST /api/ai_requirement/run-stream/`  
**视图**：`backend/apps/ai_requirement/views.py` → `run_stream_view`

| 项目 | 实现 |
|------|------|
| 流式来源 | `RequirementAgentRouter.run_stream()`（或 `run_dual_model_verified`），内部 `async for chunk in stream` 逐 token 从 LLM 拉取 |
| event_stream | `async def event_stream()`，`async for event in gen` 中 `evt_type in ('chunk', ...)` 时 `yield f"data: {json.dumps({'type': 'chunk', 'content': event['content']}, ...)}\n\n"` 后 **`await asyncio.sleep(0)`** |
| yield 后让出 | **有** `await asyncio.sleep(0)`（便于 Daphne/ASGI 立即发送） |
| 响应头 | `Cache-Control: no-cache`、`X-Accel-Buffering: no`，**未设置** `Content-Encoding: identity` |

---

## 二、关键差异小结（后端）

| 差异项 | 用例生成 | 需求智能体 |
|--------|----------|------------|
| **Content-Encoding** | **identity**（禁用 GZip，避免缓冲） | **未设置**（可能被 GZipMiddleware 压缩并缓冲） |
| yield 后 asyncio.sleep(0) | 无 | 有 |

**结论**：  
- 项目已启用 `django.middleware.gzip.GZipMiddleware`（见 `backend/backend/settings.py`）。  
- 对未设置 `Content-Encoding: identity` 的响应，中间件会对流式响应进行压缩，导致**先缓冲再压缩再发送**，前端无法收到逐 chunk 的 SSE。  
- 用例生成在所有 SSE 流式接口上显式设置了 `Content-Encoding: identity`，因此流式可见；需求智能体未设置，流式很可能被 GZip 缓冲掉。

## 三、前端对比

### 1. 用例生成（AiTestcaseGenerator.vue + api.js）

**API**：`aiGenerateTestcaseStream(data, onChunk, onDone, onError, onStart)`

- `fetch` 直连 `sseBase + '/api/ai_testcase/generate-stream/'`（开发环境直连 8009，绕过 Vite 代理）。
- `response.body.getReader()` + `while (true) { const { done, value } = await reader.read(); ... }` 按 TCP chunk 读取。
- 按行解析 `data: {...}`，`event.type === 'chunk'` 时直接 **`onChunk(event.content)`**，**无** `requestAnimationFrame`。

**页面 onChunk**：

```js
(content) => {
    streamContent.value += content
    nextTick(() => {
        if (streamOutputRef.value) streamOutputRef.value.scrollTop = ...
    })
}
```

### 2. 需求智能体（AiRequirementAgent.vue + api.js）

**API**：`aiRequirementStream(data, onChunk, onDone, onError, onStart, onCostEstimateRequired)`

- `fetch` 直连 `sseBase + '/api/ai_requirement/run-stream/'`。
- 同样 `getReader()` + `reader.read()` 循环，按行解析 SSE。
- `event.type === 'chunk'` 时 **`onChunk(event.content)`**，然后 **`await new Promise(r => requestAnimationFrame(r))`**（等待下一帧再处理下一 chunk，便于重绘）。

**页面 onChunk**：

```js
(content) => {
    streamContent.value += content
    nextTick(() => {
        requestAnimationFrame(() => {
            if (streamOutputRef.value) streamOutputRef.value.scrollTop = ...
        })
    })
}
```

### 前端差异小结

| 差异项 | 用例生成 | 需求智能体 |
|--------|----------|------------|
| API 层 onChunk 后 | 无 requestAnimationFrame | 有 requestAnimationFrame |
| 页面 onChunk | nextTick + scroll | nextTick + requestAnimationFrame + scroll |
| 流式区域占位 | `v-if="generating \|\| !currentResult"` 的 stream-output，无 min-height | stream-output 有 min-height: 120px + 「等待输出…」 |

需求智能体在前端做了更多「利于流式展示」的处理（rAF、占位、min-height），但若后端因 GZip 缓冲导致迟迟才收到第一个 chunk，前端再优化也看不到逐字效果。

---

## 四、建议修改（需求智能体）

1. **与用例生成对齐**：在需求智能体所有返回 `text/event-stream` 的 `StreamingHttpResponse` 上增加：
   - `response['Content-Encoding'] = 'identity'`
   这样 GZipMiddleware 不会压缩该响应，避免缓冲，流式即可逐 chunk 到达前端。

2. 保持现有 `await asyncio.sleep(0)` 与前端 requestAnimationFrame 逻辑不变，作为补充优化。

修改位置建议：
- `run_stream_view` 的 `StreamingHttpResponse` 后
- `clarify_and_continue_view` 的流式响应后
- `chat_stream_view` 的流式响应后
- 其他返回 SSE 的视图（如工作流、多智能体等）若也希望流式可见，同样加上 `Content-Encoding: identity`。
